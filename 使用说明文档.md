## 一、系统概述

本系统是一套面向企业内网的 Python 自动化数据备份框架，为用友 U8 ERP（SQL Server）、多台 MySQL 主机（含 Docker Percona 模式和原生二进制模式）、SQLite 数据库以及远程文件服务器提供统一的备份、校验、推送与恢复验证能力。所有依赖已离线打包，无需访问互联网。

### 核心特性

| 特性 | 说明 |
|------|------|
| 多数据库类型 | SQL Server、MySQL（Docker/原生）、SQLite、远程文件 |
| 远程连接 | SSH 密码认证（Linux）、WinRM NTLM（Windows）、WinRM 失败自动回退 SSH |
| 读写分离备份 | SQL Server 支持 `read_replica`，从只读副本执行备份，主库零压力 |
| 多盘轮询 | 自动扫描本地和远程 Windows 机器上容量 ≥ 2 TB 且剩余 ≥ 100 GB 的磁盘 |
| MD5 双重校验 | 本地 `.md5` 伴生文件 + 推送到远程存储后的远程侧 PowerShell MD5 复验 |
| 备份链审计与熔断 | 每次增量备份前审计本周备份链，发现缺口自动降级为全量 |
| 月度恢复验证 | 每月指定日自动将全量备份还原到测试机，查询表数量并告警结果 |
| 监控面板 | Apple 风格 Web Dashboard，含磁盘、时间线、备份链、告警四图 |
| 企业内网离线部署 | 全部依赖（paramiko、pywinrm 等）预置于 `vendor/` 目录 |
| 热加载配置 | 持续调度循环每分钟重读 `config.json`，修改配置无需重启进程 |

### 架构数据流

```
┌─────────────────────────────────────────────────────────────┐
│                      数据源（备份对象）                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐   │
│  │ SQL Server   │  │ MySQL 主机   │  │ Linux/Windows    │   │
│  │ 192.168.2.50 │  │ 138 / OA 等  │  │ 远程文件服务器    │   │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘   │
└─────────┼─────────────────┼───────────────────┼─────────────┘
          │ sqlcmd BACKUP   │ SSH + mysqldump    │ SSH/WinRM + tar/Compress-Archive
          ▼                 ▼                    ▼
┌─────────────────────────────────────────────────────────────┐
│                  调度机（本机 Windows Server）                  │
│                                                             │
│  scheduler → executor → validator(MD5) → alerter            │
│                    ↓                                        │
│             本地备份目录（backup_dir）                          │
│          D:\backups\u8\  D:\backups\mysql_138\  ...         │
└──────────────────────────┬──────────────────────────────────┘
                           │ push_file_to_remote (WinRM/SSH + MD5复验)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              远程存储机（backup_target Windows Server）         │
│                                                             │
│  自动多盘轮询（≥2TB，剩余≥100GB）                               │
│  E:\Backups\u8\  E:\Backups\mysql_138\  ...                 │
└─────────────────────────────────────────────────────────────┘
                           │（每月指定日）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              恢复验证测试机                                     │
│  Linux 测试机（MySQL 验证）  +  Windows 测试机（SQL Server 验证）│
│  还原 → 查表数 → 结果告警                                       │
└─────────────────────────────────────────────────────────────┘
```

## 二、系统要求

### 调度机（运行本系统的 Windows Server）

| 项目 | 要求 |
|------|------|
| 操作系统 | Windows Server 2016 / 2019 / 2022 |
| Python | 3.11 或更高版本（推荐配合 `uv` 管理虚拟环境） |
| `uv` 工具 | 用于创建虚拟环境与离线安装依赖，需预先安装 |
| `sqlcmd` | 已安装 SQL Server 命令行工具（备份 SQL Server 时必须） |
| 网络 | 能访问各数据源主机及远程存储机（企业内网，无需互联网） |

### 数据源主机

| 类型 | 要求 |
|------|------|
| SQL Server 源机 | sqlcmd 可连接；若配置 `read_replica` 则调度机直接连只读副本 |
| MySQL 源机（SSH） | 开放 22 端口；mysqldump 可用（原生路径）或 Docker 已有 Percona 镜像 |
| Linux 文件源 | 开放 22 端口，SSH 密码认证，tar 命令可用 |
| Windows 文件源 | 开放 5985（WinRM）或 22（SSH）端口，域账号认证 |

### 远程存储机

- Windows Server，开放 WinRM（5985）或 SSH（22）端口
- 至少一块容量 ≥ `min_size_tb`（默认 2 TB）、剩余 ≥ `min_free_gb`（默认 100 GB）的磁盘

### Python 依赖（已离线打包）

| 包 | 用途 |
|----|------|
| `paramiko` | SSH 连接、SFTP 文件传输 |
| `pywinrm` | WinRM NTLM 认证、PowerShell 远程执行 |

## 三、安装部署

### 步骤 1：解压发布包

将 `backup-system-release.zip` 解压到调度机上的目标目录，例如 `D:\backup-system\`。

### 步骤 2：运行 install.bat

```bat
install.bat
```

`install.bat` 依次执行三步：

1. `uv venv .venv` — 创建隔离的 Python 虚拟环境
2. `uv pip install --no-index --find-links vendor -r requirements.txt` — 从 `vendor/` 目录离线安装全部依赖，**不访问互联网**
3. 输出提示，建议运行 `start.bat --dry-run` 验证安装

```
[Backup System Installer]
[1/3] Creating virtual environment...
[2/3] Installing dependencies (offline)...
[3/3] Done! Run start.bat --dry-run to verify.
```

> 若 `uv` 未安装，请先从内网镜像获取并安装，或向管理员申请。

### 步骤 3：创建配置文件

```bat
copy config.example.json config.json
```

用文本编辑器打开 `config.json`，按第四章说明填写各项配置，将所有 `<your-password>` 占位符替换为实际密码。

### 步骤 4：验证配置

```bat
start.bat --dry-run
```

干运行模式会打印今日调度计划，不执行任何备份，也不写日志文件，可安全用于验证配置是否正确解析。

### 步骤 5：首次手动运行

```bat
start.bat --once
```

执行一次全部任务后退出。首次运行所有任务均为全量备份（无历史记录）。

### 4.1 databases[] — 数据库备份目标

`databases` 是一个数组，支持任意数量的条目，每个条目代表一个独立的备份任务，拥有独立的调度时间和备份目录。

#### SQL Server 配置

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 任务唯一名称，用于日志、状态文件和备份文件命名 |
| `type` | string | 是 | 固定填 `"sqlserver"` |
| `enabled` | boolean | 是 | `true` 启用，`false` 禁用（禁用的任务在 dry-run 中显示为 DISABLED） |
| `server` | string | 是 | SQL Server 主机地址或实例名（主库地址） |
| `read_replica` | string | 否 | 只读副本地址。**配置后备份从副本执行，主库零压力**，适合用友U8等读写分离架构。未配置时从 `server` 备份 |
| `database` | string | 是 | 要备份的数据库名称 |
| `auth.user` | string | 是 | SQL Server 登录用户名 |
| `auth.password` | string | 是 | SQL Server 登录密码 |
| `backup_dir` | string | 是 | 本地备份目录，例如 `D:\backups\u8` |
| `schedule_time` | string | 是 | 每日调度时间，UTC 格式 `HH:MM`，例如 `"02:00"` |

**`read_replica` 说明：** 配置只读副本后，系统在执行 `BACKUP DATABASE` 命令时将目标切换为副本地址，主库继续服务业务，备份 I/O 完全由副本承担。这对于用友U8等生产环境中主库负载较高的场景尤为重要。

**SQL Server 示例（带只读副本）：**

```json
{
  "name": "sqlserver_u8",
  "type": "sqlserver",
  "enabled": true,
  "server": "192.168.2.50",
  "read_replica": "192.168.2.51",
  "database": "UFDATA_001_2016",
  "auth": {
    "user": "sa",
    "password": "替换为实际密码"
  },
  "backup_dir": "D:\\backups\\u8",
  "schedule_time": "02:00"
}
```

备份文件命名格式：`{name}_{YYYYMMDD}_full.bak`（全量）或 `{name}_{YYYYMMDD}_diff.bak`（差异）。

#### MySQL 配置（重点）

MySQL 备份采用 SSH 连接到能运行 `mysqldump` 的主机，在远程执行 dump 命令，输出压缩后通过 SFTP 拉回本地，并进行 MD5 校验。

**连接架构：**

```
调度机 ──SSH──→ SSH主机（可运行mysqldump）──→ MySQL服务（mysql_host:mysql_port）
```

SSH 主机和 MySQL 主机可以是同一台机器（`mysql_host: "127.0.0.1"`），也可以不同（SSH 主机是一台能访问 MySQL 服务的中间机）。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `name` | string | 是 | — | 任务名称 |
| `type` | string | 是 | — | 固定填 `"mysql"` |
| `enabled` | boolean | 是 | — | 是否启用 |
| `ssh_host` | string | 是 | — | SSH 登录的目标主机 IP，必须能在该机器上运行 mysqldump |
| `ssh_port` | integer | 否 | `22` | SSH 端口 |
| `ssh_user` | string | 是 | — | SSH 登录用户名 |
| `ssh_password` | string | 是 | — | SSH 登录密码 |
| `mysql_host` | string | 否 | `"127.0.0.1"` | mysqldump 连接的 MySQL 主机地址（相对于SSH主机的地址） |
| `mysql_port` | integer | 否 | `3306` | MySQL 端口 |
| `mysql_user` | string | 是 | — | MySQL 用户名（与 SSH 账号独立） |
| `mysql_password` | string | 否 | `""` | MySQL 密码 |
| `mysqldump_path` | string | 否 | `"mysqldump"` | mysqldump 命令路径，支持两种形式（见下方说明） |
| `mysql_path` | string | 否 | `"mysql"` | mysql 命令路径（全库备份时用于列举数据库） |
| `temp_data` | string | 否 | `"/tmp"` | SSH 主机上的临时暂存目录，dump 文件先写到这里再被拉回 |
| `databases` | array | 否 | `[]` | 指定要备份的库名列表。**空数组表示备份全部库**（按 `exclude_dbs` 过滤） |
| `exclude_dbs` | array | 否 | 见下 | 全库备份时排除的库，默认排除系统库 |
| `backup_dir` | string | 是 | — | 本地备份目录 |
| `schedule_time` | string | 是 | — | 调度时间，UTC `HH:MM` |

**`exclude_dbs` 默认值：** `["mysql", "information_schema", "performance_schema", "sys"]`

**`mysqldump_path` 两种模式：**

| 模式 | 配置示例 | 适用场景 |
|------|---------|---------|
| 原生二进制 | `"/usr/bin/mysqldump"` | SSH 主机上已安装 mysqldump |
| Docker 封装 | `"docker run --rm --network host docker.hp/percona:5.7.25 mysqldump"` | SSH 主机通过 Docker 运行 Percona mysqldump，不需要本地安装 |

> Docker 模式：`mysqldump_path` 和 `mysql_path` 分别指向完整的 `docker run ...` 命令，系统会自动在后面追加连接参数和库名。Docker 镜像必须在 SSH 主机上已存在（离线预拉或内网镜像仓库）。

**示例一：Docker Percona 模式（全库备份，排除系统库）**

```json
{
  "name": "mysql_docker_138",
  "type": "mysql",
  "enabled": true,
  "ssh_host": "192.168.2.138",
  "ssh_port": 22,
  "ssh_user": "root",
  "ssh_password": "替换为SSH密码",
  "mysql_host": "127.0.0.1",
  "mysql_port": 3306,
  "mysql_user": "root",
  "mysql_password": "替换为MySQL密码",
  "mysqldump_path": "docker run --rm --network host docker.hp/percona:5.7.25 mysqldump",
  "mysql_path": "docker run --rm --network host docker.hp/percona:5.7.25 mysql",
  "temp_data": "/data/mysql",
  "databases": [],
  "exclude_dbs": ["mysql", "information_schema", "performance_schema", "sys"],
  "backup_dir": "D:\\backups\\mysql_138",
  "schedule_time": "02:30"
}
```

**示例二：原生二进制模式（只备份指定库）**

```json
{
  "name": "mysql_native_oa",
  "type": "mysql",
  "enabled": true,
  "ssh_host": "192.168.188.182",
  "ssh_port": 22,
  "ssh_user": "root",
  "ssh_password": "替换为SSH密码",
  "mysql_host": "127.0.0.1",
  "mysql_port": 3306,
  "mysql_user": "root2",
  "mysql_password": "替换为MySQL密码",
  "mysqldump_path": "/usr/bin/mysqldump",
  "mysql_path": "/usr/bin/mysql",
  "temp_data": "/data/mysql",
  "databases": ["td_oa"],
  "exclude_dbs": ["mysql", "information_schema", "performance_schema", "sys"],
  "backup_dir": "D:\\backups\\mysql_oa",
  "schedule_time": "02:45"
}
```

**向后兼容（旧格式）：** 只配置 `database`（单数）和 `mysql_user` 的旧配置仍然有效，系统会自动识别并只备份该单个库。旧配置示例：

```json
{
  "name": "mysql_legacy_compat",
  "type": "mysql",
  "enabled": false,
  "ssh_host": "192.168.1.12",
  "ssh_user": "backup",
  "ssh_password": "替换为密码",
  "database": "analytics",
  "mysql_user": "backup_user",
  "backup_dir": "D:\\backups\\mysql_analytics",
  "schedule_time": "03:00"
}
```

**mysqldump 参数说明：** 系统固定使用 `--single-transaction --routines --triggers --set-gtid-purged=OFF`，适合 InnoDB 引擎的在线一致性备份。备份文件以 gzip 压缩后存储（`.sql.gz`）。

#### SQLite 配置

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 任务名称 |
| `type` | string | 是 | 固定填 `"sqlite"` |
| `enabled` | boolean | 是 | 是否启用 |
| `db_path` | string | 是 | SQLite 数据库文件的本地路径 |

```json
  "enabled": true,
  "db_path": "C:\\app\\data\\app.db",
  "backup_dir": "D:\\backups\\sqlite",
  "schedule_time": "03:00"
}
```

### 4.2 remote_file_sources[] — 远程文件备份源

用于备份 Linux 或 Windows 远程主机上的目录（非数据库文件）。

#### Linux 文件源

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 任务名称 |
| `os` | string | 是 | 固定填 `"linux"` |
| `enabled` | boolean | 是 | 是否启用 |
| `host` | string | 是 | 远程主机 IP |
| `port` | integer | 否 | SSH 端口，默认 `22` |
| `username` | string | 是 | SSH 用户名 |
| `password` | string | 是 | SSH 密码 |
| `source_dirs` | array | 是 | 要备份的远程目录列表，例如 `["/opt/app/uploads", "/var/app/data"]` |
| `backup_dir` | string | 是 | 本地备份目录 |
| `schedule_time` | string | 是 | 调度时间，UTC `HH:MM` |

**备份方式：** 通过 SSH 在远程执行 `tar -czf` 命令压缩指定目录，生成 `.tar.gz` 文件，再通过 SFTP 下载到本地，最后执行 MD5 校验。增量备份使用 `find -newermt` 只打包上次备份后修改的文件。

```json
{
  "name": "linux_app_files",
  "os": "linux",
  "enabled": true,
  "host": "192.168.1.20",
  "port": 22,
  "username": "backup",
  "password": "替换为SSH密码",
  "source_dirs": ["/opt/app/uploads", "/var/app/data"],
  "backup_dir": "D:\\backups\\linux_files",
  "schedule_time": "04:00"
}
```

#### Windows 文件源

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 任务名称 |
| `os` | string | 是 | 固定填 `"windows"` |
| `enabled` | boolean | 是 | 是否启用 |
| `host` | string | 是 | 远程主机 IP |
| `winrm_port` | integer | 否 | WinRM 端口，默认 `5985` |
| `ssh_port` | integer | 否 | SSH 端口（WinRM 失败时回退），默认 `22` |
| `domain` | string | 否 | 域名（域账号认证时填写） |
| `username` | string | 是 | 用户名 |
| `password` | string | 是 | 密码 |
| `source_dirs` | array | 是 | 要备份的远程目录列表，例如 `["C:\\SharedDocs", "D:\\AppData"]` |
| `backup_dir` | string | 是 | 本地备份目录 |
| `schedule_time` | string | 是 | 调度时间，UTC `HH:MM` |

**备份方式：** 通过 WinRM（失败时回退 SSH）远程执行 PowerShell `Compress-Archive` 命令，将指定目录压缩为 `.zip` 文件，再下载到本地并校验 MD5。增量备份通过 `Where-Object { $_.LastWriteTime -gt $cutoff }` 筛选变更文件。

```json
{
  "name": "win_shared_docs",
  "os": "windows",
  "enabled": true,
  "host": "192.168.1.30",
  "winrm_port": 5985,
  "ssh_port": 22,
  "domain": "CORP",
  "username": "admin",
  "password": "替换为域账号密码",
  "source_dirs": ["C:\\SharedDocs", "D:\\AppData"],
  "backup_dir": "D:\\backups\\win_files",
  "schedule_time": "04:30"
}
```

### 4.3 backup_target — 远程存储服务器

配置后，每次备份成功后系统会自动将本地备份文件推送到这台远程 Windows 存储机，并在远程侧执行 MD5 复验。

| 字段 | 类型 | 必填 | 默认值 | 说明 |

| `winrm_port` | integer | 否 | `5985` | WinRM 端口 |

## 五、连接方式说明

系统实现了三个连接器类，分别适用于不同场景：

### 连接器对比

| 连接器类 | 适用目标 | 认证方式 | 底层实现 | 自动重试 |
|---------|---------|---------|---------|---------|
| `SSHPasswordConnector` | Linux 主机 | 用户名 + 密码 | paramiko SSH + SFTP | 是，默认3次，间隔30秒 |
| `WinRMConnector` | Windows 主机 | 域账号 NTLM | pywinrm PowerShell | 否（单次） |
| `WindowsConnector` | Windows 主机 | 域账号 | WinRM 优先，失败自动回退 SSH | 是（WinRM失败后切SSH） |

### SSHPasswordConnector（Linux）

- 连接参数：`host`、`port`（默认22）、`username`、`password`
- 文件传输：SFTP，支持上传和下载
- 命令执行：单次命令超时 300 秒
- 适用于：MySQL 备份的 SSH 隧道、Linux 文件源备份、MySQL 恢复验证测试机

### WinRMConnector（Windows 直连）

- 连接参数：`host`、`winrm_port`（默认5985）、`domain`、`username`、`password`
- 协议：HTTP WinRM + NTLM 认证，执行 PowerShell 脚本
- 文件传输：Base64 编码分块（50KB/块）通过 PowerShell 写入，适合中小文件
- 磁盘查询：通过 `Get-WmiObject Win32_LogicalDisk` 获取各盘信息

### WindowsConnector（WinRM + SSH 回退）

连接流程：

```
尝试 WinRM → 执行 echo 测试 → 成功? 使用 WinRM
                                   否? 尝试 SSH → 成功? 使用 SSH
                                                      否? 抛出 ConnectionError
```

一旦连接协议确定，整个会话保持该协议，不会再切换。适用于：远程存储机、Windows 文件源、SQL Server 恢复验证测试机。

### 数据库连接 vs 存储机连接

两种连接服务于不同用途：

**数据库备份流程（MySQL 示例）：**
```
调度机 ─SSH→ 源主机 → 执行 mysqldump | gzip → 暂存到 temp_data
           ← SFTP 下载 .sql.gz ←
           → 本地 MD5 校验
           → push_file_to_remote 推送到存储机
```

**存储机推送流程：**
```
调度机 ─WinRM/SSH→ 存储机 → PowerShell 扫描磁盘 → 选择目标盘
                          ← 上传文件 ←
                          → PowerShell Get-FileHash → 远程MD5复验
```

## 六、运行使用

### CLI 命令参考

所有命令通过 `start.bat` 调用，后面的参数直接透传给 `src/main.py`。

| 命令 | 说明 |
|------|------|
| `start.bat --dry-run` | 打印今日调度计划（全量或增量），显示禁用任务，不执行任何操作 |
| `start.bat --dry-run --job sqlserver_u8` | 只查看指定任务的调度计划 |
| `start.bat --once` | 执行一次全部已启用任务后退出 |
| `start.bat --once --job mysql_docker_138` | 只执行指定名称的任务（按 `name` 字段匹配） |
| `start.bat --verify-restore` | 立即触发月度恢复验证（不受 `schedule_day` 限制） |
| `start.bat --config other.json` | 使用指定配置文件（默认 `config.json`） |
| `start.bat` | 进入持续调度循环，每分钟检查一次，按 `schedule_time` 触发任务 |
| `start_dashboard.bat` | 启动 Web 监控面板（http://127.0.0.1:8080） |

### 持续调度模式详解

运行 `start.bat`（无参数）进入持续调度循环：

- 每 60 秒检查一次，将当前 UTC 时间（`HH:MM`）与各任务的 `schedule_time` 对比
- 匹配时触发对应任务，执行完成后继续等待
- 自动热加载 `config.json`，每分钟重读一次，修改配置立即生效（无需重启）
- 通过 PID 文件（`backup.pid`）防止同一目录下的多实例运行
- `Ctrl+C` 优雅退出，自动释放 PID 锁

## 八、备份链与熔断机制

### 备份链结构

本系统采用「周为单位」的备份链：

```
周日（D+0）  周一（D+1）  周二（D+2）  周三（D+3）  周四（D+4）  周五（D+5）  周六（D+6）
  全量          增量          增量          增量          增量          增量          增量
  *_full.*      *_incr.*      *_incr.*      *_incr.*      *_incr.*      *_incr.*      *_incr.*
  （SQL Server 增量后缀为 *_diff.*）
```

### MD5 伴生文件

每个备份文件旁边生成同名的 `.md5` 文件，例如：

```
D:\backups\u8\sqlserver_u8_20260613_full.bak
D:\backups\u8\sqlserver_u8_20260613_full.bak.md5
```

`.md5` 文件内容为该备份文件的十六进制 MD5 值。推送到远程存储时，通过 PowerShell `Get-FileHash` 再次计算远程文件 MD5 并与本地对比，确保传输完整性。

### MD5 验证流程

```
backup_result → validate_backup_file()
    ├── 文件存在？
    ├── 文件大小 > 0？
    ├── .md5 伴生文件存在？
    └── 计算实际 MD5 == .md5 中记录的值？
            失败 → 自动重试一次备份
                      仍失败 → alert_md5_mismatch() → 发送告警邮件
```

### 备份链审计（每次增量前执行）

`audit_backup_chain()` 扫描 `backup_dir` 中本周的预期文件列表：

1. 检查周日全量文件是否存在且非空
2. 检查周一至昨天的所有增量文件是否存在且非空
3. 返回 `missing_files` 列表（缺失日期的 `YYYYMMDD` 字符串）

### 熔断降级

```
audit_backup_chain() 发现缺口
    → fuse_check() 返回 "force_full"
        → 覆盖 backup_type = "full"
        → alert_chain_broken() 发送告警
        → 执行全量备份重建链
```

熔断不会中止任务，而是自动降级为全量备份，确保备份链在异常后能自动恢复。

## 九、月度恢复验证

### 触发方式

两种方式均可触发：

1. **自动触发：** `restore_test.enabled: true` 且 `--once` 运行当天的日期等于 `schedule_day`
2. **手动触发：** 随时执行 `start.bat --verify-restore`

### MySQL 恢复验证流程

```
1. 扫描本地 databases[] 中第一个 mysql 类型任务的 backup_dir
2. 找到最新的 *_full.sql.gz 文件
3. SSH 连接 linux_machine 测试机
4. SFTP 上传 .sql.gz 文件到 /tmp/restore_test_*
5. 执行: DROP DATABASE IF EXISTS `test_db`; CREATE DATABASE `test_db`;
6. 执行: gunzip < 文件 | mysql test_db
7. 查询: SELECT COUNT(*) FROM information_schema.tables WHERE table_schema='test_db'
8. 清理远程临时文件
9. 记录 tables_count，通过 alerter 发送成功/失败告警
```

### SQL Server 恢复验证流程

```
1. 扫描本地 databases[] 中第一个 sqlserver 类型任务的 backup_dir
2. 找到最新的 *_full.bak 文件
3. WindowsConnector 连接 windows_machine 测试机（WinRM 或 SSH）
4. 确保远程 C:\Temp\ 目录存在
5. 上传 .bak 文件到测试机 C:\Temp\restore_test_*.bak
6. sqlcmd 执行: RESTORE DATABASE [test_db] FROM DISK='...' WITH REPLACE, RECOVERY
7. sqlcmd 执行: SELECT COUNT(*) FROM [test_db].sys.tables
8. 清理远程 .bak 文件
9. 记录 tables_count，通过 alerter 发送成功/失败告警
```

## 十、故障排查

### 常见问题速查

| 现象 | 可能原因 | 排查步骤 |
|------|---------|---------|
| SSH 连接失败（MySQL / Linux 文件） | `ssh_host` 错误、端口不通、密码错误 | 确认 `ssh_host`/`ssh_port` 可访问，检查防火墙是否放行22端口，核对 `ssh_password` |
| WinRM 连接失败（Windows 文件 / 存储机） | WinRM 未启用、5985端口不通、域账号问题 | 在目标机运行 `Enable-PSRemoting -Force`，确认5985端口开放；系统会自动回退SSH，检查SSH是否可用 |
| mysqldump 失败：镜像不存在 | Docker 模式下镜像未拉取 | 在 SSH 主机执行 `docker images | grep percona`，确认镜像存在；如不存在从内网镜像仓库拉取 |
| mysqldump 失败：Access denied | `mysql_user`/`mysql_password` 错误 | 在 SSH 主机手动执行 `mysql -u<用户> -p<密码> -h<mysql_host> -e "SHOW DATABASES"` 验证 |
| sqlcmd 未找到 | 调度机未安装 SQL Server 工具 | 安装 SQL Server 命令行工具，确保 `sqlcmd` 在 PATH 中 |
| 磁盘空间不足 / 没有可用目标盘 | 所有磁盘不满足 `min_size_tb`/`min_free_gb` 条件 | 检查 `disks` 和 `backup_target` 配置中的阈值，确认目标机上确实存在满足条件的磁盘 |
| MD5 不匹配 | 网络传输中断或文件损坏 | 系统会自动重试一次；持续失败请检查网络稳定性，查看日志确认具体的 expected 和 actual MD5 |
| 邮件发送失败 | SMTP 配置错误或服务器不可达 | 不影响备份任务。检查 `smtp_host`/`smtp_port`，确认 `enabled: true`，查看日志中的具体 SMTP 错误 |
| 备份链断裂告警 | 某天的备份文件缺失 | 系统会自动熔断降级为全量备份。查看日志确认是哪天的文件缺失，检查那天的备份是否曾经失败 |
| Another backup instance is already running | PID 锁文件残留 | 检查 `backup.pid` 中的 PID 是否实际存活（`tasklist /FI "PID eq <pid>"`）；若进程已死则删除 `backup.pid` 文件 |
| restore_test 找不到备份文件 | 本地 backup_dir 中无全量备份 | 先运行 `start.bat --once` 确保存在 `*_full.bak`/`*_full.sql.gz` 文件 |

### 日志查看

日志文件位于 `logs/backup.log`，每天零点自动滚动，格式：

```
[2026-06-13 02:00:15 UTC] [INFO] Starting full backup for job=sqlserver_u8 → D:\backups\u8
[2026-06-13 02:03:44 UTC] [INFO] Backup OK: job=sqlserver_u8 file=..._full.bak md5=a3f4...
[2026-06-13 02:03:45 UTC] [INFO] _push_to_remote_target: D:\backups\u8\..._full.bak → E:\Backups\sqlserver_u8\... OK
```

错误级别日志还会同时触发告警邮件（若 `email.enabled: true`）。

```
│   ├── validator.py        # MD5 校验，备份链审计，熔断决策
│   ├── alerter.py          # 告警分发：日志 + 邮件
│   ├── connector.py        # 远程连接层：SSH/WinRM/WindowsConnector
│   ├── restore_verifier.py # 月度恢复验证：MySQL + SQL Server

├── logs/                   # 运行日志（自动创建）
├── config.example.json     # 配置模板，包含全部字段示例
├── config.json             # 实际配置文件（自建，不入版本库）
├── backup_status.json      # 运行状态文件（自动生成，供 Dashboard 读取）
├── backup.pid              # PID 锁文件（运行时自动管理）

├── start_dashboard.bat     # 启动监控面板

└── pyproject.toml          # 项目元数据
```

### 各模块职责详解

| 模块 | 核心职责 |
|------|---------|
| `main.py` | CLI 入口，串联所有模块；无业务逻辑，纯编排 |
| `config.py` | 从 JSON 文件加载配置，基础格式校验 |
| `scheduler.py` | `should_run_full_backup()` / `get_today_schedule()` / `run_schedule_loop()`；决定每个任务今天是全量还是增量 |
| `executor.py` | `dispatch_backup()` 根据 `type` 路由到具体实现；`backup_sqlserver`、`backup_mysql`、`backup_sqlite`、`backup_remote_files` |
| `validator.py` | `validate_backup_file()` 五项校验（路径、存在、大小、MD5伴生、哈希匹配）；`audit_backup_chain()` 扫描本周备份链 |
| `alerter.py` | `send_alert()` 统一告警出口；`setup_logger()` 配置滚动日志；具名告警模板函数 |
| `connector.py` | `SSHPasswordConnector`、`WinRMConnector`、`WindowsConnector`；`create_connector()` 工厂函数 |
| `restore_verifier.py` | `verify_mysql_restore()` / `verify_sqlserver_restore()` / `run_monthly_verification()` |
| `utils.py` | `scan_large_disks()` 磁盘枚举；`get_target_disk()` 磁盘选择；`push_file_to_remote()` 推送+MD5复验；`acquire_lock()`/`release_lock()` PID锁；`read_status()`/`write_status()` 原子读写 |
| `run_server.py` | Dashboard 轻量 Web 服务，读取 `backup_status.json` 提供 API |
