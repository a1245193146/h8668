以下是一个数据备份系统的零散需求，备份类型有sqlserver,mysql,sqllite,文件
统一调度：Python 调度程序每天凌晨触发。针对 Linux (MySQL)：通过 SSH 触发 Linux 本地脚本，备份并压缩后，将文件通过 SFTP 拉回 Windows 备份服务器。针对 Windows (SQL Server)：直接通过 subprocess 调用 sqlcmd 执行本地备份，直接输出到当前可用的大磁盘。多盘动态轮询（>2TB 自动接力）：Python 自动扫描服务器上所有挂载的盘符，筛选出容量 $> 2\text{TB}$ 的磁盘。备份时优先写入第一个盘，当其剩余空间不足时，自动、无缝地切换到下一个大磁盘继续备份。
数据校验：落盘后立即进行 MD5 校验和备份链完整性审计。
可视化展示：前端轻量级网页读取备份目录结构，动态渲染成可视化图表。
整个 Python 程序分为四个核心模块：调度器、执行器、校验器、告警器。
1. 配置管理 (config.py)
将数据库连接信息、备份路径、备份策略结构化存储。
2. 核心控制逻辑 (main.py)
利用 Python 的时间判断，决定今天跑全量还是增量。
三、 数据完整性与可用性保障机制（核心重点）
Python 的优势在于能做复杂的逻辑判断。为了防止“备份链断裂”，程序必须包含以下多重防御：

1. 物理完整性：文件级别的 MD5 强校验
在 Linux 和 Windows 本地脚本生成备份文件后，必须同时在本地生成一个 .md5 文件。
当 Python 程序将文件同步到 NAS 集中存储后，由 Python 读取该文件并在存储服务器端重新计算一次 MD5。
逻辑：对比本地生成的 MD5 与存储端的 MD5。如果一致，说明传输过程中没有发生任何因网络波动导致的丢包或损坏；如果不一致，立即触发报警并重新同步。
2. 逻辑完整性：备份链连续性自动断点检查
在执行增量备份前，Python 必须扮演“审计员”的角色：
周一至周六（增量日）：程序在启动当天增量备份前，先去扫描存储目录，检查本周日的那个全备文件是否安然无存？前几天的增量文件是否大小大于0？
熔断机制：如果发现本周二的增量文件丢了，而今天是周三，Python 应当触发熔断，放弃今天的增量备份，强行降级并立刻触发一次全量备份。因为中间断了，后面的增量全废了。通过这种强行补全备的方式，为后续恢复留住一条活路。

自动建库：在测试服务器上，Python 连接到测试 MySQL/SQL Server，创建一个 test_restore_db。
流水线还原：Python 自动读取上周日的物理全备进行还原，还原成功后，根据文件名排序，依次把周一到周六的增量/日志文件用 SQL 语句灌进去。
